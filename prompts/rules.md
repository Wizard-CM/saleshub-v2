### Deciding where a component belongs

Important Context Note : We are talking about : `app/components` folder 

- **`ui/`**: Auto-generated by the shadcn CLI. Contains only shadcn primitives. Do not manually create files here.
- **`common/`**: Any reusable component you build yourself — domain-agnostic and used across multiple features or pages. **Always leverage `shadcn/ui` primitives from the `ui/` folder as the foundational building blocks for these components before writing custom UI from scratch.**
- **`templates/<page-name>/`**: All components that belong to a specific page. The `index.tsx` file is the main page layout; every other file in the folder is a component used by that page.
- **`templates/<functionality>/`**: When components are grouped by a shared functionality that spans multiple routes (e.g. `auth`, `onboarding`, `checkout`), group them under a functionality name instead of a page name. No `index.tsx` is required in this case — each file is a standalone component for that flow.

### The rule of thumb

> If a custom component is used only within one page or one feature flow, it lives inside `templates/`. If it is reused across unrelated pages, it moves to `common/`.

## 4. Layout & Shared Components

Place components that appear on every page (navbar, footer, sidebar) in the root layout file so they render once and persist across route changes.

## 5. Comments

Add brief comments that explain *why* something exists or *why* a non-obvious decision was made. Avoid comments that merely restate what the code does.

```tsx
// Good — explains intent
// Debounce resize handler to avoid layout thrashing on low-end devices
useEffect(() => { ... }, []);

// Bad — restates the code
// Set count to 0
setCount(0);
```

## 6. General Standards

### Prefer shadcn/ui for Reusable Components 

- **Do not reinvent the wheel.** Before building a new reusable UI element (e.g., dropdowns, modals, tabs) from scratch, always use the `shadcn/ui` library. Install the required primitive via the shadcn CLI (which will place it in `ui/`), and compose or wrap those primitives to build your custom reusable components inside the `common/` directory.

### Consistent spacing
- All pages must share the same outer padding and content max-width. Define these as Tailwind CSS variables 

### Inherit over duplicate
- When multiple sibling elements need the same style (text color, font size, gap, etc.), apply it once on the parent and let children inherit. Only override at the child level when a specific child genuinely differs.

### Component size
- Keep components small and single-purpose. 

### Co-location
- Co-locate tests, types, and constants alongside their component when they are feature-specific.

### Exports
- Use named exports for components and types to improve refactoring and search.

### State management
- Prefer composition over prop-drilling; reach for context or a state manager only when data genuinely needs to cross multiple levels.

## 7. Responsiveness

Every component must be fully responsive across three breakpoints at minimum:

| Breakpoint | Target           | Tailwind prefix |
|------------|------------------|-----------------|
| Default    | Mobile (< 640px) | *(none)* |
| `md`       | Tablet (≥ 768px) | `md:`           |
| `lg`       | Laptop (≥ 1024px)| `lg:`           |

- **Build mobile-first**: write base styles for the smallest screen, then layer on `md:` and `lg:` overrides.
- Verify that no content overflows, truncates unexpectedly, or becomes unreachable at any width between 320px and 1440px.
- Common patterns to check: navigation collapsing to a hamburger menu, grid columns reducing, horizontal padding scaling down, font sizes adjusting for readability.
